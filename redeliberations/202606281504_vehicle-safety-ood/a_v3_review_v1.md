# OOD 设计方案审查报告（v1）

## 审查结果

APPROVED

## 逐维度审查

### 1. 类型系统可行性

**[通过]** 设计方案中的类型形态选择与仓颉类型系统能力完全匹配：

- **聚合根/实体** 使用 `class`（引用类型，具备独立标识和生命周期），符合仓颉单继承约束——四个聚合根（Trip、Driver、Vehicle、SystemAccount）均为平级独立 class，无多层继承关系，不存在违反单继承限制的设计。
- **值对象** 统一使用 `struct`（值类型，值相等语义），与仓颉 `struct` 的天然值相等行为一致。VO-03 处"若因实现约束使用 class 则需重写相等性方法"的设计约束表明设计者对仓颉 class/struct 语义差异有准确把握。
- **有限分类** 统一使用 `enum`（RiskLevel、AlertType、SensorStatus、AccountRole、StatusColor），均为简单枚举成员，与仓颉枚举能力匹配。
- **领域服务与端口** 使用 `interface` 定义行为契约，仓颉支持多接口实现，符合依赖倒置原则。
- **泛型使用** 设计中涉及的 `Result<T, E>`、`Option<T>`、`Array<T>` 均为标准泛型模式，在仓颉泛型系统能力范围内。
- **聚合间引用** 一律使用标识（ID），在仓颉中以 `struct` 类型表达标识型——轻量、不可变、栈分配，合理利用仓颉值类型能力。

**[轻微]** 设计中多次提到"编译期穷尽检查"作为使用 `enum + match` 的动机（如 VO-01、DS-07 二维映射）。此能力在仓颉中是否原生支持取决于 `match` 表达式的穷尽性检查策略。若仓颉的 `match` 对枚举未做到编译期穷尽可能需要借助 `else` 分支兜底，属于实现细节，不影响设计可行性。

### 2. 标准库与生态覆盖

**[通过]** 设计对仓颉标准库能力的假设合理：

- 设计中依赖的基础集合类型（`Array<T>` 等）在仓颉标准库覆盖范围内。
- `Result<T, E>` 和 `Option<T>` 作为错误处理与可选值表达的基础类型，仓颉标准库具备等价构造。
- 设计方案明确承认"仓颉标准库未原生提供领域事件发布/订阅机制，需自定义实现"（§6.2），并在决策 10 中将事件总线分为边缘侧同步回调与云端侧异步投递两条链路——该自实现方案是架构级合理决策，不构成阻塞。
- outbox 模式（§5.3、§6.2）为应用级模式，需自定义实现，设计已将其归入基础设施层职责，不要求标准库原生支持。

**[轻微]** 设计方案在 §6.1 云侧部署描述中引用"Spring Boot 服务部署"，而设计目标语言为仓颉。Spring Boot 是 Java 框架，若需以仓颉实现云侧服务，需仓颉侧的 Web 服务框架（或通过 FFI 与 Java 互操作）。此为部署层面的技术选型细节，不影响领域层架构设计的合理性。

### 3. 语言特性可行性

**[通过]** 设计所采用的错误处理、并发与资源管理策略与仓颉语言能力兼容：

- **错误处理**（§五）：采用 `Option<T>` / `Result<T, E>` 模式表达"可能无结果"与"业务失败"语义，与仓颉的 Option/Result 类型体系一致。三类错误（A/B/C）的分类与处理策略合理——A 类返回 None、B 类使用 Result、C 类向上抛异常。
- **并发设计**（§六）：边缘侧单线程流水线处理（确定性实时响应），云端侧无状态领域服务 + 乐观锁聚合根并发控制——该模型在仓颉中可实施。边缘侧无并发写冲突的设计简化了并发复杂度。
- **资源管理**：outbox 模式将事件持久化与聚合根状态更新置于同一事务，保证"状态变更"与"事件已发布"一致性——该模式在仓颉中通过显式资源管理实现。
- **模块/包结构**：`domain.model`、`domain.risk`、`domain.event` 等模块划分符合 cjpm 包组织方式，依赖方向单向（`domain.model` 为最底层，上层模块依赖下层），无循环依赖。

**[轻微]** 设计中"事件持久化与聚合根状态更新在同一事务中提交"（§5.3）隐含了对跨存储（如事件表与聚合根可能存储在不同介质）的事务支持。在具体实现中若两者存储介质不同，需借助 outbox 表作为中介。此为实现细节，架构层面的事务边界设计是合理的。

### 4. 设计一致性

**[通过]** 各抽象的职责描述清晰，协作关系形成完整闭环：

- **聚合根职责明确**：Trip 为感知数据与告警汇聚中心，Driver 管理身份与健康档案，Vehicle 管理设备状态，SystemAccount 管理账户与权限。四者间通过标识引用，边界清晰。
- **判定模型划分一致**：流式融合判定（门面+委托，仅 FATIGUE/DISTRACTION/ROAD_RAGE）与事件触发型判定（独立服务，LIFE_DETECTION/COLLISION_DISABILITY）的区分在 DS-01、DS-05、DS-06、决策 2、决策 13 中反复确认，无自相矛盾。
- **风险解除语义闭环**：RiskResolvedEvent 作为独立事件类型表达风险解除，VO-02 明确其不污染 AlertType/RiskLevel 取值，DS-01/DS-04/DS-07/DS-08/DS-13 的消费链形成完整闭环。场景 3（路怒解除→停止录制→恢复空调）与场景 7（风险解除→权限撤销）的端到端路径可追踪。
- **数据链路闭合**：BR-05 评分公式的四类输入（重度疲劳、分心、路怒、急刹/急加速）均有明确来源——前三者来源于 Trip 关联的 SafetyAlertEvent 统计，后者来源于 Trip 持有的 DrivingBehaviorCounters（VO-16）→ DS-17 全程计数→ 基础设施层加速度监测。Driver 综合风险评分的计算与写回链路（DS-09 → DriverScoreUpdatedEvent → Driver 聚合）在 AR-02、DS-09、场景 5 步骤 6 中完全闭合。
- **二次身份验证门控**：DS-08 的授权流程中明确作为门控点，VO-07 补充约束说明，场景 11 显式加入步骤，错误表（§5.2）新增 SecondaryAuthRequired——安全规则在设计层面可追踪。
- **模块依赖方向**：`domain.model` 无外部依赖 → `domain.event` 仅依赖 model → 各领域服务模块依赖 model + event，模块间禁止直接调用、通过事件解耦。依赖关系有向无环。
- **16 个领域服务均有场景覆盖**：场景 1-12 覆盖全部领域服务的核心交互路径。

**[轻微]** VO-12 InterventionInstruction 的"指令类型建议以 enum 表达"未给出完整的指令类型枚举成员清单（仅示例性列举：氛围灯调色/语音播报/座椅震动/双闪/空调调节/音频播放/CAN 减速请求/引导靠边），在详细设计阶段需补充完整枚举值。当前架构级设计的抽象程度下此不构成一致性问题。

### 5. 设计质量

**[通过]** 设计在职责划分、抽象层次和可测试性方面质量良好：

- **单一职责**：各领域服务职责聚焦——FatigueDeterminationService 仅负责疲劳判定、DistractionDetectionService 仅负责分心检出、ScoringService 仅负责评分计算等，无职责混杂。
- **抽象层次恰当**：本方案为架构级 OOD，聚焦领域对象的类型形态、职责边界和协作契约。未过早涉入具体字段定义、方法签名等实现细节，抽象层次匹配设计阶段目标。
- **可测试性**：
  - 领域服务设计为无状态（DS-01 门面通过会话上下文注入状态以保持纯函数语义、DS-05 以 DetectionWindow 值对象承载窗口状态），给定输入确定输出，便于单元测试。
  - 领域服务与基础设施依赖通过 `interface` 端口解耦（VehicleStateBuffer、DrivingBehaviorTrackingPort），可 mock 实现，符合依赖倒置原则。
  - 聚合根独立（通过标识引用），可独立加载和测试。
- **设计可演进**：流式判定门面+委托模式使各判定规则可独立演进，`interface` 定义的行为契约允许不同判定算法互替换（边缘轻量规则 vs 云端 AI 模型）。
- **修订追溯清晰**：a_v3_v1 的修订说明表逐项对应诊断意见与修改措施，从 a_v2_v1 到 a_v3_v1 的演进路径完整可审计。

**[轻微]** DS-03 DistractionDetectionService 的 0.5s 端到端时延约束（判定成立到告警下发）在设计中通过"边缘侧同步消费链路"给出实施策略，但未明确说明若同步链路阻塞（如 InterventionService 生成指令耗时过长）时的超时/降级策略。此为详细设计阶段可补充的健壮性考量，不阻塞架构通过。

## 修改要求（REJECTED 时存在）

无。审查级别问题为零，设计方案在仓颉语言可行性方面无阻塞性缺陷。
