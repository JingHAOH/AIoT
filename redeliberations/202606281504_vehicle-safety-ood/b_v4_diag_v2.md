## 一、总体评价

产出在经四轮迭代修复后整体质量较高：BR-01~BR-08 全部业务规则均有对应领域对象覆盖，req_v4 §五的八项核心业务对象均已被建模为聚合根/实体/值对象，16 个领域服务（含 DS-18）均有行为契约场景。以下问题集中在**内部逻辑一致性矛盾**、**结构性格式错误**、**关键组件/端口接口缺失**三个方面，修复后产出可进入编码实现阶段。

## 二、问题清单

### 问题 1【严重】§3.5 领域事件章节存在结构错误——DS-18 领域服务被误置于事件章节下，且章节标题重复

- **问题描述**：文档第 613 行出现 `### 3.5 领域事件` 标题，但其下内容为 **DS-18 DriverScoreUpdateService（领域服务）**（第 617 行），而非领域事件。第 627 行再次出现 `### 3.5 领域事件` 标题，下方才是真正的领域事件一览表。这导致：(a) 文档结构混乱——领域服务被错误归类到领域事件章节下；(b) 同一章节号 `3.5` 在文档中出现两次，违反文档编号唯一性；(c) 下游实现者在按章节号引用时将产生歧义（"§3.5 领域事件"到底指 DS-18 还是事件表？）。
- **所在位置**：第 613 行（DS-18 及其 `### 3.5 领域事件` 标题）与第 627 行（重复的 `### 3.5 领域事件` 标题）
- **严重程度**：严重（阻塞下游编码——编号唯一性缺失导致引用歧义）
- **改进建议**：(a) 将 DS-18 DriverScoreUpdateService 移至 §3.4 领域服务章节末尾，作为 DS-18；(b) 删除第 613 行的 `### 3.5 领域事件` 标题；(c) 确保 §3.5 标题仅在第 627 行出现一次。

### 问题 2【严重】Trip:Driver=1:1 基数约束与 L3DurationTracker 的多 Driver key 映射假设在设计内部多处自相矛盾

- **问题描述**（v2 已修正归因）：AR-01 Trip 第 78 行声明"一个 Trip 属于一个 Driver 和一个 Vehicle"（Trip:Driver = 1:1），该约束与 req_v4 §五第 144 行"行驶行程…隶属一名驾驶员与一台车辆"一致。但设计在**多处**（**含 AR-01 自身**）引入了与该 1:1 约束相悖的多 Driver 假设：
  - AR-01 第 81 行："内部持有 L3DurationTracker（VO-17）值对象（**按 Driver 标识做 key**）……用于追踪 **per-Driver** 的 L3 持续时长"；
  - VO-17 第 364 行："一个 Trip 内**每个被监测的 Driver** 最多存在一个活跃的 L3DurationTracker"；第 371 行："**多 Driver 并发判定**时，各 Driver 的 L3 状态由各自的 Tracker 独立追踪，Trip 聚合内**以 Driver 标识做 key 映射**"；
  - DS-08 第 489 行复用同一"各 Driver 的 Tracker 以 Driver 标识做 key 隔离"表述。
  既然一个 Trip 仅隶属一名 Driver，则 Trip 聚合内的 per-Driver key 映射与"多 Driver 并发判定"在结构上不可能发生，属冗余且与 1:1 基数约束直接冲突。**注意：此矛盾并非"AR-01 正确、仅 VO-17/DS-08 错误"——AR-01 第 81 行自身即参与了该矛盾**（v1 报告对此处归因有误，已据质询更正）。
- **所在位置**：AR-01 第 78 行（Trip:Driver=1:1）⇔ AR-01 第 81 行、VO-17 第 364/371 行、DS-08 第 489 行（per-Driver key 映射 / 多 Driver 并发判定）
- **严重程度**：严重（逻辑矛盾——在同一设计内对同一概念给出互斥约束，实现者无法确定以哪个为准；且矛盾分散于聚合根、值对象、领域服务三处）
- **改进建议**：**须统一上述全部位置的基数约束**，而非仅改其中一处——
  - 方案 (a)（推荐，与 req_v4 一致）：维持 Trip:Driver=1:1，从 **AR-01 第 81 行、VO-17 第 364/371 行、DS-08 第 489 行**一并移除"按 Driver 标识做 key / per-Driver / 多 Driver 并发判定 / 以 Driver 标识做 key 隔离"等表述，改为"Trip 聚合直接持有单一 L3DurationTracker"的简单持有语义；
  - 方案 (b)：若确有未来多 Driver（如换手驾驶、同程多人监测）扩展意图，则须在 AR-01 第 78 行对 Trip:Driver 基数显式放宽并说明理由，且与 req_v4 §五达成一致，再保留 key 映射。
  无论取哪一方案，AR-01 第 78 行与第 81 行、VO-17、DS-08 必须表述一致。

### 问题 3【一般】领域事件总线仅声明"需自定义实现"，未给出设计契约

- **问题描述**：§6.2 声明"仓颉标准库未原生提供领域事件发布/订阅机制，需自定义实现"，并区分为边缘侧同步消费（进程内回调）和云端异步消费（outbox + 消息队列）。但设计文档始终未定义这个自定义事件总线的**接口契约**——事件注册/发布/订阅的 API 形态、线程模型、错误语义、outbox 的投递保证级别等均缺失。所有领域服务的协作说明均以"订阅某事件""产出某事件"描述，但这些描述依赖一个尚未定义的事件总线接口，领域服务的具体实现（订阅注册方式、事件承载类型、同步/异步分发路由）因此在设计层面悬空。
- **所在位置**：§6.2（第 884–887 行）、§3.4 各领域服务的"协作"小节、决策 10
- **严重程度**：一般（不阻塞理解，但阻塞编码——所有事件发布/订阅代码无接口契约可依）
- **改进建议**：在 §3.5 或独立设计决策中定义领域事件总线的核心接口契约，至少包含：(a) 事件发布接口签名（如 `publish(event: DomainEvent): Unit`）；(b) 同步消费注册机制（边缘侧，如 `registerSyncHandler<T>(handler)`）；(c) 异步消费的生命周期管理；(d) outbox 表结构与投递保证级别。可参照仓颉现有模式撰写。

### 问题 4【一般】"会话上下文"被多处引用且持有内容已列举，但从未升格为正式建模的领域组件

- **问题描述**（v2 已修正措辞）：DS-01 RiskDeterminationService 依赖"边缘侧流式会话上下文"持有当前活跃风险集（第 383 行）；DS-16 DriverStatusBroadcastService 依赖"边缘侧会话上下文的当前活跃风险集"（第 593 行）；DS-05 LifeDetectionService 依赖"调用上下文（边缘侧的活体监测会话）"持有 DetectionWindow（第 434 行）。三处均引用同一个组件——"会话上下文"。**其持有内容已在 §6.2 第 880 行列举**（"流式会话上下文（当前活跃风险集、DetectionWindow、L3DurationTracker）亦由该单线程持有"），故"持有内容未建模"的判断不成立。但该组件**仍未被升格为正式建模的领域对象/组件**：其生命周期（何时创建/销毁）、与 Trip 聚合的关系（是 Trip 聚合的运行时投影/缓存，还是独立于 Trip 的组件——尤其 L3DurationTracker 一面声明归 Trip 聚合持有、一面又说归会话上下文持有，二者归属关系未厘清）、线程安全语义，均未在任何章节中正式定义。多个核心服务依赖一个仅在并发说明里"顺带提及"、未正式建模的组件，编码时可能被重复实现或误实现。
- **所在位置**：DS-01（第 383 行）、DS-16（第 593 行）、DS-05（第 434 行）；持有内容见 §6.2 第 880 行；与 Trip 聚合的归属张力见 VO-17（第 364–371 行）、DS-08（第 489 行）
- **严重程度**：一般（多组件依赖同一未正式建模的概念，且其与 Trip 聚合的持有关系存在表述张力，编码时易误实现）
- **改进建议**：(a) 在 §三 或决策章节新增对"边缘侧会话上下文"的正式定义，明确其为 Trip 聚合的运行时投影/缓存还是独立组件；(b) 厘清其持有内容（当前活跃风险集、DetectionWindow、L3DurationTracker）与 Trip 聚合的关系——同一内容（尤其 L3DurationTracker）究竟归会话上下文持有还是归 Trip 聚合持有，须二者一致、不得双重归属；(c) 明确其生命周期边界（随 Trip 创建/销毁，还是独立于 Trip）。

### 问题 5【一般】外部系统（SMN/SparkRTC/IoTDA/120 救援中心/OTA 下发）的领域层端口契约缺失

- **问题描述**（v2 已补充 OTA 下发通道）：需求明确技术栈包含 SMN（推送）、SparkRTC（音视频对讲）、IoTDA（MQTT 设备接入）。OOD 中多项领域服务依赖这些外部系统——DS-16 依赖推送通道、DS-12 向救援中心/120 上报 RescueReport、家属对讲/视频依赖 SparkRTC、**DS-15 OTA 升级包下发依赖 IoTDA/MQTT 通道**——但它们全部以自然语言描述（如"经基础设施层推送通道下发""由基础设施层投递至救援中心""升级包下发"），未声明领域层的依赖端口接口。对比之下，VehicleStateBuffer（决策 14）和 DrivingBehaviorTrackingPort（决策 17）均有完整端口契约（方法签名、参数类型、返回类型、错误语义），而外部推送/音视频/救援上报/OTA 下发的端口却全部缺失，下游无法并行开发。
- **所在位置**：DS-12（第 540–546 行）、DS-16（第 593–597 行）、DS-15（第 577–584 行）、§6.2 跨端推送（第 889 行）
- **严重程度**：一般（阻塞上下游并行开发——基础设施层不知道领域层需要什么接口）
- **改进建议**：参照决策 14/17 的端口建模方式，为以下外部依赖定义端口契约（至少含方法签名与错误语义）：(a) **NotificationPort**（推送，替代 SMN 直连——`sendNotification(target: AccountId, payload: NotificationPayload): Result<Unit, PushError>`）；(b) **RescueReportPort**（救援上报——`submitReport(report: RescueReport): Result<Unit, ReportError>`）；(c) **MediaSessionPort**（音视频对讲通道建立与拆除——核心方法的输入输出与错误语义）；(d) **OTADeliveryPort**（OTA 升级包下发——经 IoTDA/MQTT 的分片下发/断点续传偏移量提交/校验结果回报的方法契约与错误语义）。如认为这些端口属基础设施层而不应在领域层定义，需在决策中**统一说明理由并描画替代的协作机制**（不可对 VehicleStateBuffer/DrivingBehaviorTrackingPort 建端口、对上述外部依赖却以自然语言带过，标准应一致）。

### 问题 6【轻微】急刹/急加速的检出阈值未定义来源

- **问题描述**：DS-17 DrivingBehaviorTrackingService 依赖基础设施层的加速度监测组件按阈值检出急刹/急加速事件，VO-16 DrivingBehaviorCounters 亦依此递增。但急刹/急加速的加速度阈值（减速度大于何值？加速度大于何值？）在整个 OOD 中从未被声明——既未作为配置参数出现，也未作为 DrivingBehaviorTrackingPort 的端口契约参数（第 354 行仅写"超过阈值"而无数值，决策 17 端口未将阈值列为参数）。阈值是领域规则（影响"什么算急刹"从而影响评分公式输入）还是纯基础设施调优参数，设计未予区分，实现者将自行杜撰阈值。
- **所在位置**：VO-16 DrivingBehaviorCounters（第 346–358 行）、DS-17（第 601–609 行）、决策 17
- **严重程度**：轻微（不影响架构理解，但阈值缺失将导致不同实现者产生不同判定结果，影响评分一致性）
- **改进建议**：(a) 在 VO-16 或 DrivingBehaviorTrackingPort 契约中声明急刹/急加速阈值作为端口参数（或领域配置项），使检出逻辑有统一标准；(b) 或者明确标注阈值为基础设施层配置，领域层不关心具体数值，但需注明"不同阈值将导致评分不一致，验收时需固定阈值"。

### 问题 7【轻微】救援机构（AccountRole.RESCUE？）角色未建模

- **问题描述**（v2 已补需求落点）：req_v4 **§二第 31 行**将系统参与者明确为"五类参与者"，**第 41 行**列"救援机构：外部应急响应方"为其中之一；req_v4 **§3.6 第 100–101 行**明确其权限（第 100 行"远程解锁授权：救援机构核实险情后…授权开启车门锁"、第 101 行"基础健康档案调取…救援机构在授权后可调取"）。但 OOD 中 AR-04 SystemAccount 的 AccountRole（VO-14，第 323 行）仅含 FAMILY 和 MANAGER 两个枚举值，救援机构无对应角色定义。DS-12 EmergencyRescueService 中对救援机构的授权校验仅以自然语言描述（"救援机构核实险情后云端授权开启车门锁""经授权后可调取"），缺少角色建模和权限校验的类型安全分支。
- **所在位置**：VO-14 AccountRole（第 323 行，仅 FAMILY/MANAGER）、AR-04 SystemAccount（§3.1）、DS-12（第 540 行）
- **严重程度**：轻微（救援机构可能不需要持久化的 SystemAccount 实体——其授权可能是短期令牌——但设计应明确说明这一假设，而非仅以枚举缺值暗示）
- **改进建议**：(a) 如救援机构确需建模为 SystemAccount，则为 AccountRole 增加 RESCUE 枚举值并在 DS-12 中实现类型安全的权限分支；(b) 如救援机构的授权走临时令牌/一次性授权而非持久账户（更符合实际），则在 DS-12 或决策中明确说明此假设，以消除"角色枚举缺值是否为遗漏"的歧义。

### 问题 8【轻微】家属通知偏好被引用但未形式化为可配置字段/值对象（缺风险等级粒度）

- **问题描述**（v2 已修正前提事实）：req_v4 §3.4 **第 83 行**明确"分级预警推送：家属可设置并接收**轻/中/重度不同等级**的状态异常预警"，这意味着每个家属账户应有可配置、可按风险等级订阅的通知偏好。OOD 中 AR-04 **第 129 行已引用**"接收 SafetyAlertEvent 并**根据角色（AccountRole）与通知偏好决定是否推送**"、第 122 行职责列有"通知权限"——因此设计**并非"全量推送"**（v1 报告"全量推送"的前提与原文相悖，已据质询更正）。但"通知偏好"仅作为**概念被引用**，从未形式化为可落地的字段/值对象：AR-04 与 VO-07 Permission 均未包含通知偏好字段或关联值对象，缺少 `Set<RiskLevel>` 这样的**风险等级订阅粒度**，DS-16 与告警推送链路也未体现"按家属偏好按等级过滤后推送"的可执行契约。即"按偏好推送"有声明、无可实现的过滤契约，与需求"可设置并接收不同等级"之间存在落地空白。
- **所在位置**：AR-04 SystemAccount（第 122、129 行）、VO-07 Permission（§3.3）、DS-16 与告警推送相关协作描述
- **严重程度**：轻微（可通过在 AR-04 中追加 `NotificationPreference` 值对象解决，不影响整体架构）
- **改进建议**：在 AR-04 SystemAccount 或关联值对象中将"通知偏好"形式化为可配置字段/值对象（如 `NotificationPreference` 值对象，包含订阅的风险等级集合 `Set<RiskLevel>`），并在 DS-16 和告警推送协作说明中补充"按家属偏好（含风险等级粒度）过滤后推送"的可落地语义，使第 129 行引用的"通知偏好"有具体类型支撑。

- **先前迭代反馈修复**：迭代第 1~3 轮的全部问题均已在 a_v4 中修复，未发现回退或遗漏。
- **BR-05 评分数据全链路**：传感器 → 加速度监测 → 急刹/急加速检出 → DrivingBehaviorCounters → ScoringService 的完整链路已通过 VO-16、DS-17、决策 17 闭合（数值阈值缺失见问题 6）。
- **二次身份验证建模**：DS-08 门控约束、场景 11 二次验证步骤、决策 18 设计决策均已到位且与 req_v4 §六（第 169 行）一致。
- **隐私边界（BR-04）**：DMS 脱敏、路怒语音存证边缘侧存储/加密/到期清除、RoadRageVoiceRecord 不上云等约束均已在领域对象和事件中建模。
