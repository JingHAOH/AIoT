# OOD 设计方案审查报告（a_v4_review_v1）

## 审查结果

APPROVED

## 逐维度审查

### 1. 类型系统可行性

**[通过]** 设计中所有类型形态选择均与仓颉类型系统能力匹配：

- 聚合根（AR-01~04）和实体（E-01~03）使用 `class`，仓颉 `class` 为引用类型，支持单继承和多接口实现，设计中不存在违反单继承约束的继承层次。
- 值对象（VO-03~05、VO-07~13、VO-15~17）统一使用 `struct`，仓颉 `struct` 为值类型且天然具备值相等语义，与值对象"由属性值定义相等性"的核心语义完全一致。各值对象之间无继承关系，不涉及 `struct` 不支持继承的限制。
- 有限分类体系（VO-01 RiskLevel、VO-02 AlertType、VO-06 SensorStatus、VO-14 AccountRole、VO-15 StatusColor、VO-12 指令类型）使用 `enum`，仓颉 `enum` 支持 `match` 编译期穷尽检查，与设计中多处"穷尽检查""类型安全分支"的约束一致。
- 端口抽象（VehicleStateBuffer、DrivingBehaviorTrackingPort、子判定服务接口）使用 `interface`，仓颉 `interface` 可定义方法契约，支持在装配阶段注入实现，与领域层的端口-适配器模式兼容。
- 泛型使用 `Array<T>`、`Option<T>`、`Result<T, E>`，均在仓颉泛型系统能力范围内，无协变/逆变等复杂泛型约束诉求。
- 协作关系中的类型交互（聚合根持有值对象集合、领域服务以值对象为输入输出）均为仓颉支持的基本类型组合模式。

### 2. 标准库与生态覆盖

**[通过]** 设计所需的核心能力均在仓颉标准库或合理自定义范围内：

- `Option<T>`（表达"可能无结果"的语义）、`Result<T, E>`（表达"可能成功或业务失败"的语义）均为仓颉标准库类型，设计与仓颉错误处理惯例一致。
- `Array<T>` 及集合类型用于聚合根内持有值对象集合，仓颉标准库提供。
- `match` 模式匹配用于枚举分支，仓颉语言原生支持。
- 领域事件发布/订阅机制：设计在 §6.2 明确声明"仓颉标准库未原生提供领域事件发布/订阅机制，需自定义实现"，并给出了边缘侧同步回调 + 云端侧 outbox+消息队列的分层实现策略。作为架构级设计，准确识别标准库能力边界并提出可行替代方案是恰当的，不构成设计缺陷。
- cjpm 项目组织方式：模块结构 `domain.model` / `domain.risk` / `domain.intervention` 等可直接映射为仓颉 package 层级，与 cjpm 的项目目录约定兼容。

**[轻微]** 领域事件总线的自定义实现策略在 §6.2 已充分描述，但未显式声明该自定义组件应作为独立的 `infrastructure.event` 或类似基础设施模块存在。建议在设计层面补充一句："自定义事件总线（含同步回调注册与 outbox 投递器）作为基础设施层独立模块，领域层仅依赖其接口"，以保持端口-适配器分层清晰。此建议不影响当前设计的可行性。

### 3. 语言特性可行性

**[通过]** 设计中的各语言特性使用策略均与仓颉能力匹配：

- 错误处理策略（§五）：`Option<T>` / `Result<T, E>` 的错误分类（A/B/C 三类）与仓颉的错误处理惯例一致。领域服务优先使用 `Option` / `Result` 而非抛异常，仅 C 类系统级故障向上抛，符合仓颉语言错误处理最佳实践。
- 并发设计（§六）：边缘侧单线程串行处理（无并发竞争）、云端侧乐观锁（聚合根版本号）、领域事件 outbox 模式（事件持久化与聚合根更新同一事务），均为仓颉可实现的并发策略。设计中不依赖语言级 actor 或特殊并发原语。
- 资源管理：领域层设计不涉及底层资源管理（文件句柄、网络连接），此维度在架构级设计中无需展开。
- 模块/包结构：`domain.*` 的模块划分与单向依赖原则兼容仓颉 package 的导入规则，无循环依赖隐患。

### 4. 设计一致性

**[通过]** 各抽象的职责描述清晰，协作关系形成闭环：

- 18 个领域服务（DS-01~DS-18）各职责边界明确，无职责重叠或遗漏。流式融合判定（DS-01 门面 + DS-02/03/04 子服务）与事件触发型判定（DS-05、DS-06）的分离决策在决策 2 和决策 13 中有充分理由阐述。
- 全部领域服务在 §四均有对应的行为契约场景（场景 1~12），覆盖率 100%。含时序/性能约束的场景（分心 ≤0.5s、OTA 状态机、报告 ≤15s、常态同步 ≥1Hz/≤2s）均有可验证契约。
- §3.5 领域事件表中 16 个事件均有明确的触发时机、携带信息和消费方，无一悬空。AlertType 取值集合在三类判定事件（RiskDeterminedEvent / LifeDetectedEvent / EmergencyActivatedEvent）之间两两不相交，消除了重复判定事件歧义。
- 模块间依赖方向为单向（`domain.model` → `domain.event` → 各领域服务模块），各领域服务模块之间禁止直接调用、统一通过领域事件解耦，无循环依赖。
- §五错误处理策略覆盖全部三类错误（A 判定失败 / B 业务规则违反 / C 系统级故障），与 §6.3 并发策略的条目对应。

**[轻微]** 文档 §3.4 与 §3.5 之间存在章节编号重复：第 613 行和第 627 行均出现 `### 3.5 领域事件` 标题，DS-18 DriverScoreUpdateService 被置于两个 §3.5 标题之间。DS-18 实质属于 §3.4 领域服务，建议将其移至 DS-17 之后（第 610 行附近），并删除第 613 行的冗余 §3.5 标题，使 DS-18→§3.5 的结构为"领域服务全部列完后进入领域事件"。此问题不改变任何设计语义，仅影响文档结构清晰度。

### 5. 设计质量

**[通过]** 设计整体质量良好，符合 DDD 领域层架构设计原则：

- 单一职责原则：各领域服务职责聚焦（如 RiskDeterminationService 仅负责流式融合门面，急刹计数独立为 DS-17，评分独立为 DS-09），无"上帝服务"。聚合根职责清晰（Trip 汇聚行程级数据、Driver 管理驾驶员身份与档案、Vehicle 管理设备状态、SystemAccount 管理账户与权限）。
- 抽象层次恰当：识别并区分了"流式融合判定"与"事件触发型判定"两种判定模型（决策 13），分离了"持续缓存（ring buffer）"与"全程计数"两种基础设施职责（决策 17），将"状态性缓存"下沉到基础设施层或值对象（DetectionWindow / L3DurationTracker）使领域服务保持无状态。这些均为合理的架构级抽象决策，不存在过度设计或设计不足。
- 便于后续详细设计和实现：18 个决策记录（决策 1~18）为每个关键设计选择提供了明确的理由、替代方案分析和仓颉语言考量，接口端口（VehicleStateBuffer、DrivingBehaviorTrackingPort）的方法契约已初步定义（决策 14、决策 17），足以解除上下游并行开发阻塞。
- 便于单元测试：接口抽象的端口（VehicleStateBuffer、DrivingBehaviorTrackingPort、子判定服务接口）可 mock；领域服务保持无状态（或状态以输入参数形式传入，如 DetectionWindow、L3DurationTracker）；错误处理使用 `Option`/`Result` 而非异常，测试断言明确。设计层面的可测试性基础良好。

## 修改要求

无。审查未发现严重或一般级别问题，设计通过。
